Persisting data for docker containers:
-----------------------------------------
1. container layer: inside the container
	at the top of the image layer in the container, docjer containers offer a thin layer of read-write file system.
	life is dependent on the container lifecycle.
	if the container stops no problem, you will not loose any data
	if the container is removed, you will loose the data

2. mounts: outside the container
	a. volumes
	b. bind mounts
	c. tmpfs (temporary file system)
	e. named pipes

a. volumes: persisting data in the host machine
--------------------------------------------------
persistent storage mechanism
managed by docker

/app/feedback

1. named volumes
	command: > docker volume create feedback-volume
	command: > docker run -d --name feedback-service -p 3001:3000 -v feedback-volume:/app/feedback feedback-image:1.0.0
	this is the best practice and suggested way of persisting data

2. anonymous volumes
	command: > docker run -d --name feedback-service -p 3001:3000 -v /app/feedback feedback-image:1.0.0

	this is the best practice for some temporary data that you need during the runtime of the container app and not beyond that


use of both types of volumes:
--------------------------------
> docker run -d --name feedback-service -p 3001:3000 -v /app/temp -v feedback-volume:/app/feedback feedback-image:1.0.0

another difference between named and anonymous volumes:
-------------------------------------------------------------
any container using anonymous volume, if is started with --rm flag, the anonymous volume will be deleted as soon as you stop the container and removed.

>docker rm container-name

>docker run --rm -d --name feedback-service -p 3001:3000 -v /app/temp -v feedback-volume:/app/feedback feedback-image:1.0.0
 
it does not happen in case if the anonymous volume has been shared with another container the anonymous volume will not be deleted even if the first container (that was started with --rm flag) is stopped and removed (for which the anonymous volume was originally created)

any container using named volume, if is started with --rm flag, the volume will NOT be deleted as soon as you stop the container and removed.

d. tmpfs:
-------------
applicable for Docker with Linux
saved in the host system memory (temporary)

docker networks:
------------------------
app in a docker container

1. wants to access some site for data (outside the scope of docker environment)
ex: https://jsonplaceholder.typicode.com/todos (a RESTful API)
	nothing to do here. any container running app can access WWW

2. wants to access some service in host system
	use "host.docker.internal" instead of "localhost"

network drivers: host

mongodb://host.docker.internal:27017 instead of mongodb://localhost:27017 

3. wants to access some service running inside another docker container
	mongodb

create custom network under bridge driver
>docker network create fav-net

run mongo db image and connect to the fav-net
>docker run --rm -d --name favserver -p 3000:3000 -e DNSVALUE=mongoserver --network fav-net fav-image:2.0.0

run your app image and connect to the fav-net in order to access the mongo db container
>docker run --rm -d --name favserver -p 3000:3000 -e DNSVALUE=mongoserver --network fav-net fav-image:2.0.0

mongodb://mongoserver:27017

When you start Docker, a default bridge network (also called bridge) is created automatically, and newly-started containers connect to it unless otherwise specified. 
You can also create user-defined custom bridge networks. 
User-defined bridge networks are superior to the default bridge network.

default bridge network does not expose container name as its DNS name. you need to look up the container's IPv4 address in order to connect to that container from another container.

custom bridge network does expose container name as its DNS name. you DO NOT need to look up the container's IPv4 address in order to connect to that container from another container. you can use the name of the other container as the DNS name.
